Introducing ‘maxqda’

Andreas Blätte (andreas.blaette@uni-due.de)

2020-11-04

In text-based research, using Machine Learning (ML) very often involves annotating a set of texts. For large annotation tasks in teams, powerful server-based solutions such as INCEpTION offer specially designed and optimized solutions. Yet for more limited scenarios, a smaller solution may be a reasonable choice. The maxqda package presents an option to use MAXQDA, a leading commercial product for qualitative data analysis, for generating annotations that can be used in an ML scenario.

Using MAXQDA to prepare annotations for an ML exercise may be reasonable if the following considerations apply:

  • You think that the administrative cost to set up and maintain a server-based annotation environment are overly high.
  • You do not need specialized functionality (such as active learning, for instance) that only dedicated server-based solutions can offer.
  • You think that the time investment to acquaint and train coders to competently use a new annotation environment is high relative to the requirements of your project.
  • You have access to an institutional license of MAXQDA and using this commercial software product does not entail additional cost and obstacles to obtain the software.

These considerations may be particularly pertinent for researchers from the social sciences with a university affiliation who are on a budget: You do not necessarily have software engineers at hand and the supportive structure to quickly set up and maintain a server for a small project, students as potential coders are very often acquainted with MAXQDA already, and many universities have institutional licenses for MAXQDA, so using MAXQDA does not produce additional costs for you project. Thus integrating MAXQDA into a ML-oriented research process is an option that may be particularly relevant for a Computational Social Science (CSS) context.

Envisaged workflow - an overview

ML was not what MAXQDA developers had in mind when developing their product. The thrust of MAXQDA is to offer a specialized tool for qualitative research (Kuckartz 20190). But the flexibility of MAXQDA to handle different data formats, including HTML is the basis for the following workflow.

  • You export documents to be annotated from R as an HTML format. Invisibly, the HTML needs to wrap tokens in elements that declare token ids and character offset positions via attributes. The html() method of polmineR will do that for you if you set arguments `

  • Documents are imported into MAXQDA and are annotated according to a pre-established coding scheme.

  • After finishing the annotation, the file used by MAXQDA is evaluated using the MAXQDA class defined in the maxqda package. It returns a data.table that keeps the essential information on the annotations that have been made.

How to use MAXQDA

Generating the required HTML format

The envisaged workflow requires that documents to be annotated are imported into MAXQDA as HTML documents. As MAXQDA stores information on annotations as character offset, the distinctive feature of these HTML documents is that that they need to capture precise information about the character offset positions of tokens and token ids. This is required for matching MAXQDA annotations on the data structure of your corpus.

This information can be included in an HTML document without being visible for the user. MAXQDA will not complain about this hidden information. Essentially, each token needs to be wrapped into a ‘span’ element that defines the attributes id, offset_left and offset_right. This might look as follows:

## [1] "<span id=\"595951\" token=\"I\" class=\"fulltext\" left=\"66\" right=\"67\">I</span><span id=\"595952\" token=\"have\" class=\"fulltext\" left=\"68\" right=\"72\">have</span><span id=\"595953\" token=\"the\" class=\"fulltext\" left=\"73\" right=\"76\">the</span><span id=\"595954\" token=\"honour\" class=\"fulltext\" left=\"77\" right=\"83\">honour</span><span id=\"595955\" token=\"to\" class=\"fulltext\" left=\"84\" right=\"86\">to</span><span id=\"595956\" token=\"present\" class=\"fulltext\" left=\"87\" right=\"94\">present</span><span id=\"595957\" token=\"my\" class=\"fulltext\" left=\"95\" right=\"97\">my</span><span id=\"595958\" token=\"millennium\" class=\"fulltext\" left=\"98\" right=\"108\">millennium</span><span id=\"595959\" token=\"report\" class=\"fulltext\" left=\"109\" right=\"115\">report</span><span id=\"595960\" token=\".\" class=\"fulltext\" left=\"116\" right=\"117\">.</span>"

This HTML data format includes a lot of overhead and hidden information to represent a simple sentence such as “I have the honour to present the millenium report.”, but MAXQDA is able to handle that. And the user can leave it to a tool to generate the HTML that meets specifications. There are many ways to do that and you could develop your own tool. At this stage, the polmineR package offers the functionality to prepare what is needed out of the box.

The html() method of polmineR will do that for you. The important part is that argument cpos and charoffset both need to be set as TRUE. Save the HTML file to disk at a directory from where it can be imported into MAXQDA. If you generate a larger set of HTML files, put them in the same directory and ensure that no other files that shall no be processed are in the same directory.

In our sample code, we use a temporary directory though.

library(polmineR)

html_file <- tempfile(fileext = ".html")

corpus("REUTERS") %>%
  subset(id == 127) %>%
  html(cpos = TRUE, charoffset = TRUE) %>%
  cat(file = html_file)

Annotating documents using MAXQDA

Documents to be annotated need to be imported into MAXQDA for annotation. MAXQDA can handle HTML files without further any ado. As creating training data requires annotating a significant number of files, there will be a set of files you will import. So we reccommend to use MAXQDA’s functionality for bulk import. If you have put all HTML files into one directory, tell MAXQDA to import all files in this directory at once.

Using MAXQDA for annotating texts is straight forward. There are many training materials to learn how to use MAXQDA. An ML annotation exercise will require that coders create annotations in line with one pre-established annotation scheme. The pre-established coding scheme relevant for your project can be imported easily.

Finally: Preparing annotations for ML purposes requires hermeutic judgement. Guiding coders to make consistent judgements by a codebook is an established practice. Beyond this, keeping the annotations created simple may be a good choice from the perspective of the ML work package: MAXQDA has a functionality that goes far beyond what can be processed by ML at this stage.

Extracting annotation data from MAXQDA files

Annotation projects are stored by MAXQDA in files with an extension such as “.mx20”. The last two digits in the filename indicate the MAXQDA version used when generating the file, referring to the year of the MAXQDA release: MAXQDA uses calendar versioning. So if a MAXQDA file ends with “.mx20”, this indicates that the 2020 release of MAXQDA has been used.

This file is a SQlite database file. This is a very common database file format. There are libraries for all programming languages to handle the sqlite3 databases. The R package RSQlite offers the interface to handle the sqlite3 file generated and used by MAXQDA.

There is no official MAXQDA API that would expose a documented access to annotations. Developing the approach of the maxqda R package to extract annotations from the mx files involved guessing how the MAXQDA project database is designed. But actually the database design is very intelligible. For basic document types such as TXT and HTML, MAXQDA stores annotations in a tabular format that defines start and end character offset positions in one table of the database that is crucial for extracting annotations from the database. Another important table within the database is a table with the codings used.

The core (and almost single) functionality of the maxqda R package is to evaluate these tables, to map the annotation data on the input corpus data and to return a data.table that will present information on annotations that can be used and integrated in the subsequent stages of an R text mining workflow.

The package is a GitHub-only package at this stage - see the README.md file of the package for installation instructions. Load the package as usual.

library(maxqda)

The functionality of the package is organised by way of an R6 reference class. After loading the maxqda package, you get started to evaluate the MAXQDA sqlite3 project file by instantiating a MAXQDA (R6) class object, supplying information on the whereabouts of the MAXQDA sqlite3 project file (argument dbFilename) and the path of the directory where HTML files that had been imported into MAXQDA for annotation reside (argument htmlDir).

The package includes some sample data derived from the REUTERS corpus and a sample MAXQDA database with some (rather arbitrary) annotations. We locate these files as follows …

dbFileSample <- system.file(package = "maxqda", "extdata", "maxqda", "reuters.mx12")
htmlDirSample <- system.file(package = "maxqda", "extdata", "html", "reuters")

And we instantiate the MAXQDA class supplying this information as described.

reutersMX <- MAXQDA$new(dbFilename = dbFileSample, htmlDir = htmlDirSample)

The MAXQDA class breaks down the steps required to evaluate the mx-file into a set of methods. The magic method is $getAll() that will return a data.frame with the annotations. When evaluating MAXQDA annotations, the HTML files will be opened to extract information on character offset positions and token IDs. By default (argument verbose = TRUE), a message informs you which HTML file is processed.

annos <- reutersMX$getAll()
## ... processing: argentina
## ... processing: canada
## ... processing: kuwait
## ... processing: qatar

The data.table has the following columns:

  • left: Left corpus position of the annotation.
  • right: Right corpus position of the annotation.
  • code: The coding that has been assigned.
  • txt: The text that has been annotated.
  • document: Name of the document where the annotation occurrs.

So the result is simple and straight forward. The information on character offset positions of annotations as stored in the MAXQDA project database has been transformed into information on token IDs (left and right corpus positions). There is a set of relevant scenarios where this is all that you need to proceed to the genuine ML part of a project. Using MAXQDA for annotation is convenient, and getting information on annotations back into R is not complicated at all when you use the maxqda R package.

Discussion

The R package maxqda is handy but its approach is not universal: It requires that HTML files meeting a specification are imported into MAXQDA and it assumes that your corpus management system assigns unique IDs to tokens. This is how the Corpus Workbench (CWB) manages corpus data, as exposed to R with the polmineR package. Using polmineR, it is easy to generate the MAXQDA input data envisaged by the workflow this package assumes. If you use another toolset for analysing corpora, the way to using MAXQDA may involve writing code. This is a limitation of the approach we envisage here.

To conclude: MAXQDA is a mature commercial software product with an outstanding usability that is widely used in the social sciences. MAXQDA has become a very powerful and flexible research tool. But it’s remaining charm is that it exposes the functionality for the annotation step in a research process involving ML in a very handy fashion. If you have an institutional access to MAXQDA licenses for yourself and for your team, if you do not need the whole functionality of a server-based specialized annotation environment and if your annotation task is limited, using MAXQDA for your ML-oriented annotation task may be an option to consider. The maxqda package gives you the functionality to realise this option.
